BCM components basically use ports and connectors to interconnect components, with inbound port exposing offered component interfaces and outbound ports exposing required component interfaces. Other technologies to interconnect software entities could also be used, such as POJO references or more interestingly REST APIs interconnections. To abstract the interconnection technology, end points are introduced in BCM. End points are defined on two interfaces, mainly component interfaces, a required one seen from the client side of the interconnection and an offered one that is exposed by the server side. The idea is that end points have all the same protocol to use them to interconnect software entities, and here mainly BCM components, so that interchanging the technology should only require to change the type of end point without touching at the rest of the code in the application.

Currently, only POJO and BCM end points are implemented but a REST type of end point is planned to be added shortly. All types of end points implements the protocol, among which the method {@code getReference} must return a Java reference to be used in the client side code to call services offered by the server side. In a POJO end point, a standard Java object reference to the server is hidden by the end point. In a BCM end point, the reference used from the client side is the one of the outbound port. The BCM end point is first defined by the offered and required components interfaces but also the URI of the inbound port. Hence, a BCM end point can be created and t hen passed to the client and the server so that the protocol will create and publish the inbound port on the server side and then create, publish and connect the outbound port on the client side. Methods to clean up the client and the server side are also part of the protocol, and for BCM end point, they will disconnect, unpublish and destroy the ports.
